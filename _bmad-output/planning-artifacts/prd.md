# Product Requirements Document - bmad-test

**Author:** centron
**Date:** 2026-03-02T19:07:35+08:00

## Executive Summary

**愿景与目标对齐**
本项目旨在打造区域教育系统的智能“领航员”，核心业务为一套以“增值评价”为核心的监测模型及前端决策大屏。系统服务于区县及市级教育局管理人员，致力于打破唯分数论的传统考核机制。通过整合多维过程性指标（如学科素养、师资流动、心理健康等），动态追踪并客观评估不同生源基础学校的成长轨迹，从而赋能教育管理者实施“精准医疗式扶持”与“特色化办学导航”，最终推动区域教育质量的全面提升。

### What Makes This Special

- **“增值评价”动态追踪引擎**：系统摒弃绝对静态排名，能够根据每所学校的历史基础连贯生成“定制化预期”，公平展示薄弱学校的增量成绩，激发学校办学潜能。
- **AI 赋能的决策驾驶舱**：大屏前端不仅呈现指标数据，更在关键指标异常时主动预警，并关联成功干预案例，直接为管理者提供政策倾斜和资源调配的可执行建议。
- **高维数据整合能力**：将隐性过程数据（如五育并举指标）与显性结果数据结合，形成完整的学校发展立体画像。

## Project Classification

- **项目类型 (Project Type):** SaaS B2B / API 中心 (决策大脑大屏 + 评价模型 API)
- **所属领域 (Domain):** EdTech (教育科技) / GovTech (政务平台)
- **复杂度 (Complexity):** 高 (High) —— 涉及复杂的教育评估算法模型制定、多维数据整合，以及严格的政务系统数据隔离、隐私合规与权限管理。
- **项目背景 (Context):** 绿地项目 (Greenfield)

## Success Criteria

### User Success
- **诊断闭环打通**：区局管理者在 3 分钟内一键生成包含基础情况、增值评价与异常预警的“单校发展体检报告”。
- **决策支持有效性**：管理者直接引用系统“增值指标数据”作为资源分配（师资、资金）的客观依据。
- **数据透明互信**：80% 受评学校校长认可“增值评价”模型的客观性。

### Business Success
- **标杆落地**：系统上线 6 个月内，在西安市至少 1-2 个试点区（县）教育局完成常态化使用。
- **效率提升**：将区县教育局原本需要各科室人工汇总计算的学校学期/跨学年综合评价耗时，降低 80% 以上。
- **标杆示范效应**：形成 3 个以上基于该系统发现“薄弱点”并成功实施“精准扶持”的区域教育治理案例。

### Technical & Compliance Success
- **数据安全与隔离**：满足政务安全规范，实现区县间数据逻辑绝对隔离与脱敏，通过教育局审计。
- **高可用与并发**：期末评估流量峰值期，API 响应延迟 < 500ms，大屏流畅无卡顿。

## Product Scope

### MVP - Minimum Viable Product
- **基础评价模型引擎 (API)**：完成包含国家教育标准基础数据、学校历年成绩及部分过程性数据的接入底座；实现基于历史基线的“增值评价”核心算法计算。
- **区县级教育决策大屏**：提供区域整体教育质量概览、分类学校表现排行（侧重增值指标）、以及多维数据雷达图展示。
- **单校“体检”与红绿灯预警**：支持查看单校的各项指标得分；当某些关键指标（如某学科平均分持续两学期下滑）严重低于模型预期时，触发红灯预警。
- **基于角色的政务权限体系 (RBAC)**：市局看全市、区局看本区，符合政务数据最小可见原则。

### Growth Features (Post-MVP)
- **AI 智能干预模型**：当发现特定类型的红灯预警时，系统利用 AI 自动基于过往成功案例或政策库，生成一套“建议处方”（如建议增加心理疏导资源、或调配某个方向的名师）。
- **区县-学校双向反馈机制**：允许校长在系统内对评估结果进行数据申诉或提交整改计划反馈，形成管理闭环。

### Vision (Future)
- **宏观教育资源沙盘推演**：支持教育管理者进行“What-if”模拟（如果在某薄弱学区投入 50 名特级教师，模型预测三年后的区域均衡度变化），真正实现未来战略规划的数字化。

## User Journeys

### 旅程 1：市级教育局管理者 —— 宏观督导与区县赋能
**背景与痛点：** 王局长在市局工作，需要统筹全市 11 个区县的教育均衡发展，但往往只能在期末或年终才能拿到滞后的评估报告。
**旅程：** 年中的某天，王局长打开“市级全景决策大脑”。系统清晰地展示了各区县的“增值发展指数”热力图。他发现 A 区的增值指数突然呈现出橙色预警。他点击下钻，发现问题不在资源投入，而是 A 区的多所学校的“心理健康与五育指标”存在断崖下降。
**高潮与解决：** 王局长没有直接发文批评，而是将大屏上的诊断报告连同一份由系统智能关联分享的“全国区县级五育并举专项提升经验报告”直接一键批转给 A 区教育局，要求他们参考方案在一周内给出应对措施。市局的角色从“滞后考核”变成了“前置赋能”。

### 旅程 2：区县教育局局长 —— 异常告警与对策借鉴 (核心路径)
**背景与痛点：** 李局长管理着区内 40 所中小学，资源有限，最怕看到突然发生的严重下滑却不知道原因。
**旅程：** 周一早晨，李局长收到系统手机端的“异常告警”：实验中学的综合发展增值排名下滑了 5 位。李局长登录区县级驾驶舱，点击进入实验中学的“体检详情”。系统清晰地通过归因分析告诉他：造成此次下滑的最大影响因子 (占比 60%) 是“青年骨干教师流失率激增”。
**高潮与解决：** 李局长查看了流失名单。同时系统弹窗展示了静态的**行业经验对策库**，其中有一条类似场景的对应建议（由市局教研专家编写，并附有其他区县的解决思路）。基于对策库中的启示，李局长决定在下学期将区内 3 名特级教师轮岗至该校带教。此举极大省去了他迷茫思考对策的时间。

### 旅程 3：中学校长 —— 接收体检与双向反馈
**背景与痛点：** 赵校长管理一所生源较为薄弱的初中，以往单纯拼升学率总是排在倒数，教职工士气低落。
**旅程：** 期末评估结束后，赵校长登录了学校专属评估后台。他惊喜地发现，虽然绝对分数他校依然不高，但在“增值评价雷达图”上，他们学校的“生均进步跨度”和“体艺特长生培养”呈现了耀眼的绿色（远超模型预期）。
**高潮与解决：** 他立刻看到了区局随报告下发的表扬信和额外的专项补贴资金额度。同时，看到报告中提示“学生硬件设施满意度”不足时，赵校长直接在系统内点击“申诉与反馈”，提交了下学期更新实验室的整改计划，完成了与区局的管理闭环。

### 旅程 4：系统管理员/实施专员 —— 指标设定与经验库维护
**背景与痛点：** 作为技术实施人员，小李需要满足市局和区局不同的指标权重需求，同时维持各区县教育系统参考案例库的更新。
**旅程：** 新学年开始，市局出台了新的评价指标体系。小李登录后台管理系统，通过“指标大厅”拖拽式地调整了各种过程性数据的权重分配，并调整了针对薄弱学校的增值算法基线。
**高潮与解决：** 配置完成后，小李进行了沙盒测试，确认各区县和学校的数据只能被对应权限的领导看到。同时，他根据市教委专家提交的 Word 文档，将最新审核通过的“突发指标异常指导对策”入库，以便日后能自动呈现给触发告警的区县。系统平稳升级。

### Journey Requirements Summary
通过上述核心用户旅程，系统必须具备以下核心能力：
1. **数据可视化与钻取能力 (UI/UX)**：从市级热力图 -> 区县雷达图 -> 单校体检明细 -> 最细颗粒度因素追溯的层层下钻能力。
2. **根因分析与归因算法 (Analytics)**：系统需能够计算不同指标权重的占比，追溯出导致结果异常的最大影响因子。
3. **静态对策库/经验指导关联机制 (Knowledge Base)**：在 MVP 阶段，建立基于“异常诊断标签”与“静态专家经验/案例”的匹配关联引荐系统。
4. **多级组织架构与公文流转 (Workflow)**：支持市局向区局批转指导意见、区局向学校下发报告、学校向区局反馈申诉或整改计划的双向层级互动机制。
5. **灵活配置体系与安全边界 (Admin/Security)**：提供“指标与权重”的可配置大厅能力，同时保障区-校级多租户数据的完全逻辑隔离与 RBAC 权限控制。

## Domain-Specific Requirements

### 合规与监管要求 (Compliance & Regulatory)
- **安全等级保护**：系统整体架构、网络传输及数据存储必须严格满足**信息安全等级保护三级（等保三级）**标准，需配合完成定级备案及第三方测评机构的测评验收。
- **数据隐私与隔离**：包含学生心理健康、问卷反馈等敏感抽样数据需进行逻辑上或物理上的严格隔离处理，应用层展示必须遵守最小可见原则（RBAC）。

### 技术物理约束 (Technical Constraints)
- **部署环境**：系统必须支持**私有云物理隔离部署**，禁止核心业务系统直接暴露在公网，外网访问需通过政务规定的安全网关。
- **统一身份认证接入**：大屏及管理后台不维护独立的独立密码体系，必须无缝集成**政务钉钉或教育局指定的统一认证平台 (SSO)** 实现免登。

### 核心集成与数据流向 (Integration Requirements)
- **数据来源单一化接入**：本系统不直接向异构的业务源头系统（如各学校独立系统）索取数据，所有基础数据（升学率、基础课成绩、师资状态等）必须由前置的**“教育数据中心(Data Center)”**清洗汇总后，通过标准 API 提供给本系统。本系统处于整个数据流的下游精加工环节。

### 风险应对机制 (Risk Mitigations)
- **数据补录与异常修正兜底**：防范前置“数据中心”抽取失败或源系统数据缺失导致评价模型无法计算的问题。在系统后台必须提供高权限的**“强修正模块（手动维护入口）”**，允许管理员对断点数据、异常数据进行人工填报或修正，并全程留存修改审计日志（Audit Log）。

## SaaS B2B & API Backend Requirements

### Project-Type Overview
本项目采用 **集中式 SaaS 架构 + 标准 API 开放平台** 的混合模式。系统不仅包含面向各级教育管理者的前端决策驾驶舱，还包含底层完全由我方团队自主设计与研发的“教育增值评价算法模型”。该模型在处理前置“数据中心”的数据后，既服务于自身前端大屏，也通过标准 API 向市/区县的其他政务系统提供评价结果和静态对策输出。

### Technical Architecture Considerations

**1. 租户与隔离模式 (Tenant Model)**
- **逻辑多租户**：系统采用单实例部署，共享单一代码库和数据库基础结构。不进行复杂的物理租户拆分。
- **行级数据安全隔离**：所有业务数据表必须通过严格的组织架构字段（如 `机构ID`、`区县ID`）进行硬过滤，确保数据按权属范围严格物理隔离。

**2. 预设计算法引擎 (Algorithm Implementation)**
- 系统的核心价值在于**我方自主设计和研发的增值评价模型算法**。系统后台将独立运行基于 Python 的算法微服务，负责定时从数据中心拉取数据进行清洗、聚类和计算。
- **同类分层对比**：MVP 阶段的算法基准建立在对学校进行合理的生源基线分层上，通过同层级内的聚类对比来客观反映增值效应，具有高可解释性。
- **参数配置沙箱**：提供专门的前端页面，允许教研团队在沙箱预配置指标的权重参数，通过“试跑数据”校验排名合理性后，方可发布至生产环境的主引擎，实现了算法构建和业务系统的解耦。

### 接口与鉴权规范 (API & Auth Models)

**1. 权限矩阵系统 (RBAC Matrix)**
系统需接入统一认证平台 (SSO)，并在系统内部建立四类角色的细粒度（颗粒度差异化）鉴权视图：
- **市级超级管理员**：拥有全局最高视图，可见所有区县宏观热力图及跨区直观对比数据。
- **区县局长/管理者**：仅可见本区数据及辖下所有学校“体检明细”，不可平移查看他区，是接收静态对策的主要受众。
- **校级领导**：仅可见本校发展趋势与明细雷达图，不可见他校，拥有申诉与反馈权限。
- **第三方运维/实施专员**：拥有算法参数沙箱配置、对策文档入库、异常数据手动修正的后台配置权限，不可查阅区/校两级隐私业务数据报表。

**2. 标准 API 开放中台 (API Endpoints & Infrastructure)**
系统必须暴露一组标准的 REST接口，以输出脱敏后的统计和指标类数据：
- 采用 JWT 或 OAuth2 的 **Token 鉴权机制** 控制接入终端。
- 配置严格的高并发 **Rate Limiting (限流)** 以保护业务库免受由于第三方轮询造成的核心大屏访问性能劣化。
- 实施明确的 **API 版本控制**（例 `/api/v1/health-radar`），确保在底层增值评定指标变化时对外部系统的调用兼容。
- 输出内容**仅包含结构化且带有诊断标签的增量推演结果及对策**，严格禁止通过此接口越权回溯基础原始层敏感数据。

### Implementation Considerations
- **容错辅评机制**：大屏在呈现所有红灯及异常指标推演前，必须经过区一级的流审核或标注，即在初期仅发挥“辅助发现异常雷达”的作用。

## Project Scoping & Phased Development

### MVP Strategy & Philosophy
**MVP Approach:** 硬核基座优先，体验功能后延
针对高复杂度的区域教育评价系统，第一期 (Phase 1) 必须优先攻克数据中心无缝接入与自研增值评价算法引擎的建立，不仅要出前端大屏结果，还要交付后台支撑其计算的权重调参沙箱。

### Phase 1: 最小可行性产品 (MVP)

**核心架构节点 (Must-Have Capabilities):**
- **数据管线**：全面对接“教育数据中心”数据流 API，支持历史全量及日常增量抽取。
- **独立算法引擎服务 (Python Compute Engine)**：实现第一代“同类分层聚类”增值评价算法。
- **动态权重沙箱视图**：可视化调整模型核心因子权重并“试跑”的独立环境。
- **静态对策/经验库 (V1)**：结构化文档库，包含市局下发的常见落后指标对策建议。
- **强数据修正模块**：针对数据中心断点数据/异常数据的管理员手工纠偏录入入口并留存审计日志。

**前端大屏与用户旅程 (Core User Journeys):**
- **单点登录与数据隔离**：集成政务 SSO 统一认证，实现严格的市/区/校三级行级别数据隔离可视化。
- **全要素驾驶舱 (Dashboard)**：支持“市级热力图 -> 区县排名 -> 单校雷达图”层层下钻。
- **基础红绿灯预警**：指标剧烈恶化时系统自动亮灯并追溯最大影响因子，并在界面提示可供参考的对策库链接。

### Post-MVP Roadmap

**Phase 2: 闭环与自动化 (Growth Phase)**
- **在线申诉与复核审批流**：校级可发起含佐证的红灯申诉；区局增设审批下发工作台。
- **强关联对策精准推送**：系统化分析降级红灯原因，直接弹框推送最匹配的经验文档供管理员批转。

**Phase 3: 开放中台与未来愿景 (Expansion Phase)**
- **标准对外 API 开放平台**：正式对外开放带 JWT/OAuth 鉴权且限流保护的脱敏指标及评估画像 REST API。
- **AI 动态沙盘推演模型**：引入变量假设推演机制预测未来区域增值指数变化。

## Functional Requirements

### 1. 认证与角色权限 (Authentication & Authorization)
- **FR1:** 用户可以通过政务钉钉或统一 SSO 门户进行免密单点登录。
- **FR2:** 系统可根据登录用户的账号属性，自动赋予市级管理员、区县管理者、校级领导或运维专员的角色。
- **FR3:** 系统可基于用户角色和其绑定的行政区划 ID，对所有业务数据界面进行硬性过滤（行级数据隔离）。

### 2. 数据集成与补录 (Data Pipeline & Fallback)
- **FR4:** 系统可定时通过 API 从“教育大数据中心”自动抽取全量历史数据及增量业务数据。
- **FR5:** 运维专员可在一个受限的高级表单内，手工录入或修改由数据中心抽取失败的缺失/错误测评数据。
- **FR5.1:** 运维专员可通过下载标准模板批量导入修正断点/异常数据，系统须在正式入库前提供数据逻辑格式的自动校验与预检报告。
- **FR6:** 系统可自动记录所有手工录入与修改操作的详细审计日志（操作人、时间、修改前/后数值）。

### 3. VAM 增值评价算法沙箱 (Algorithm Sandbox)
- **FR7:** 具有权限的教研员可在后台“沙箱”界面配置影响增值评价模型（同类分层聚类）的各项指标基准和权重得分。
- **FR8:** 具有权限的教研员可点击“试跑”，在隔离环境中预览按照当前配比权重算出的全市/全区模拟测试排名。
- **FR9:** 试跑结果审核通过后，高权限管理员可将沙箱中的配置方案一键发布，同步覆盖至生产环境算法引擎。

### 4. 评价仪表盘展示 (Evaluation Dashboard)
- **FR10:** 市级管理员可查看全覆盖的宏观“区域红绿灯热力图”，直观对比各区县的综合增值态势。
- **FR10.1:** 评价仪表盘在大屏展示各区校排名与指标时，必须显著标识该维度数据的“手工介入/人工修正比例”，提供数据置信度标签及下钻查看修改日志的入口。
- **FR11:** 区县管理者可查看全区雷达图，并向下钻取，展示辖区内每一所学校的“体检明细”和排名变化。
- **FR12:** 校级领导仅能查看本校在各项指标（如学业、身心、师资等）上的长效发展轨迹曲线及雷达画像。
- **FR12.1:** 校级领导在查看本校雷达图与预警时，系统必须提供当前“同类分层”的规则说明页面，以及该层级基线的匿名化统计特征展示（只看同类分位线，不看他校具体数据）。
- **FR13:** 当某项关键指标大幅恶化时，系统可在前端界面亮起显著的“红灯预警”标识。
- **FR14:** 系统可在亮起红灯预警的同时，自动标示出导致该综合指数下降的“最大单一负面影响因子”（如近视率飙升），并且**必须同时输出该归因算法的置信度评分（或要求教研员进行人工打标确认）**，以防范黑盒误判。

### 5. 静态经验智库 (Static Knowledge Base)
- **FR15:** 运维专员或市级管理员可在后台富文本编辑器中录入、分类、修改经验指导文档（对策库）。
- **FR16:** 区县管理者或校级领导在查看指标预警时，系统可展示指向对应问题（如：心理健康对策篇）的静态文档快捷链接文本。

### 6. 标准 API 开放中台 (API Open Space)
- **FR17:** 系统可通过标准 RESTful 接口向经过认证（Token鉴权）的外部政务平台输出计算完成的区校指标排名。
- **FR18:** API 网关可执行严格的 Rate Limiting 限流策略，拦截恶意的高频轮询调用。

## Non-Functional Requirements

### 1. 安全与合规性 (Security & Compliance)
- **NFR1:** 所有存储于数据库中的个体敏感数据必须经过国密算法加密（如 SM4）物理存储。
- **NFR2:** 系统前端或对外 API 展示结果时，不允许反查个源身份明细，需脱敏或匿名聚合输出。
- **NFR3:** 系统建设必须满足国家《信息安全技术网络安全等级保护基本要求》的“第三级”标准。
- **NFR4:** 系统必须对运维的所有数据纠偏及调参动作留存不可篡改的时间戳审计日志。

### 2. 性能与响应 (Performance)
- **NFR5:** 在市/区级管理者查看宏观热力图并发起下钻切片操作时，前端页面的首屏全量渲染时间不得超过 3秒。
- **NFR6:** 独立增值评价引擎（微服务）在沙箱接收权强调参配置后，处理覆盖 150 所学校规模的全量试算任务，返回全区新排名的计算等待时长不得超过 30分钟。

### 3. 可靠性与可用性 (Reliability)
- **NFR7:** 系统核心业务层（不含跑批沙箱）总体可用性至少需达到 99.9%。
- **NFR8:** 前置“数据中心”拉取任务中断超过 10s需判定超时并启用自动断点续传机制；连续抽取失败 3 次触发独立告警。

### 4. 接口规范与拓展性 (Integration & API)
- **NFR9:** 面向外部系统开放的脱敏 REST API 接口，在承受 100 QPS 峰值并发时，请求处理的 P95 响应时间必须小于 500ms。
- **NFR10:** 对外开放评估接口时需执行严格兼容的显式版本控制策略（Versioning），以应对自身底层模型算法调参前后的变更。
